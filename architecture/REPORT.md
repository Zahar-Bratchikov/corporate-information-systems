## ИС УПЗ IT — Информационная система управления проектами и задачами в IT

## 1. Постановка задачи

### 1.1. Описание назначения КИС и основных функциональных требований согласно ТЗ

**Назначение системы:** Программное изделие предназначено для IT-подразделений организаций: планирование и учёт IT-проектов (разработка и внедрение ПО, сервисов, доработки), управление задачами (фичи, исправления, технические задачи), распределение нагрузки между разработчиками и тестировщиками, контроль сроков и формирование отчётности для руководства и стейкхолдеров.

**Объект использования:** IT-направление организации. Пользователи — руководители проектов и тимлиды, разработчики (backend, frontend, fullstack), тестировщики (QA), аналитики, администраторы системы.

**Основные функциональные требования по ТЗ:**

| №   | Группа требований            | Содержание                                                                                                                                                                                                                                                                                                                                                                 |
| --- | ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Редактирование данных**    | Создание, изменение и удаление: IT-проекты (наименование, код, ответственный, даты релиза); задачи (название, тип — фича/баг/задача, приоритет, срок, статус); исполнители (пользователи и роли); справочники статусов и приоритетов; спринты/итерации. Назначение исполнителей на задачи, привязка задач к проектам и спринтам. Валидация с выдачей сообщений об ошибках. |
| 2   | **Пять отчётов с экспортом** | 1) Сводный по IT-проектам (Excel, PDF); 2) Задачи исполнителя с итогами (Excel, Word, PDF); 3) Просроченные задачи с вычисляемым полем «дни просрочки» и сводкой по проектам (Excel, PDF); 4) Загрузка команды по статусам, с фильтром по спринту (Excel, Word); 5) Статусы IT-проектов — агрегаты по задачам и % выполнения (Word, PDF).                                  |
| 3   | **Авторизованный доступ**    | Роли: Администратор (полный доступ), Руководитель проекта/Тимлид (управление проектами и задачами, отчёты по своим проектам и команде), Разработчик/QA (просмотр и изменение своих задач), Наблюдатель (только просмотр и отчёты в рамках видимости). Вход по логину и паролю; завершение сессии по таймауту неактивности или выходу.                                      |
| 4   | **Обработка исключений**     | При ошибочном вводе — отказ в сохранении и сообщение с указанием поля; при несанкционированном доступе — отказ и при повторных попытках фиксация в журнале; при ошибках БД/внешних сервисов — общее сообщение пользователю, детали в лог, при возможности откат транзакции; при ошибках формирования отчётов — сообщение и рекомендация повторить, фиксация в логах.       |
| 5   | **Развёртывание**            | Развёртывание в Docker-контейнерах: отдельные контейнеры для приложения и базы данных.                                                                                                                                                                                                                                                                                     |

## 2. Диаграмма вариантов использования (use case)

**[Диаграмма вариантов использования](use-case.png)**

### 2.1. Основные акторы

- **Администратор** — входит в систему, выходит, просматривает данные; управляет пользователями и справочниками (статусы, приоритеты); управляет проектами, задачами и спринтами; формирует все отчёты.
- **Руководитель проекта / Тимлид** — входит, выходит, просматривает проекты и задачи; управляет проектами, задачами и спринтами (в рамках своих зон ответственности); формирует отчёты.
- **Разработчик / QA** — входит, выходит, просматривает проекты и задачи; редактирует только свои задачи (назначения, привязки к проектам и спринтам); формирует отчёты (в т.ч. по задачам исполнителя).
- **Наблюдатель** — входит, выходит, просматривает проекты и задачи в рамках видимости; формирует отчёты; не изменяет данные.

### 2.2. Покрытие прецедентов

Прецеденты на диаграмме:

- **Пакет «Авторизация»:** «Вход в систему» (логин/пароль), «Выход из системы» (или завершение сессии по таймауту). Доступны всем ролям.
- **Пакет «Работа с данными»:** «Просмотр проектов и задач» (в рамках видимости по ролям); «Управление пользователями и справочниками» (только Администратор); «Управление проектами, задачами и спринтами» (Администратор, Руководитель/Тимлид); «Редактирование своих задач» (Разработчик/QA).
- **Пакет «Отчёты»:** «Формирование отчётов (Word, Excel, PDF)» — объединяет пять отчётов по ТЗ: сводный по IT-проектам, задачи исполнителя с итогами, просроченные задачи, загрузка команды, статусы IT-проектов. Доступно всем ролям в пределах их прав (Наблюдатель — только просмотр и отчёты).

## 3. Диаграмма системного контекста

**[Диаграмма системного контекста](system_context.png)**

### 3.1. Краткое описание основных элементов

**Акторы (Person):**

- **Администратор** — настройка системы, управление пользователями, ролями и справочниками; работа с системой через веб-интерфейс (HTTPS).
- **Руководитель проекта / Тимлид** — планирование и контроль проектов, управление задачами, формирование отчётов.
- **Разработчик / QA** — просмотр и обновление своих задач.
- **Наблюдатель** — просмотр статусов проектов и задач, формирование отчётов без изменения данных.

**Система:**

- **ИС УПЗ IT (System)** — проектируемая система в целом: планирование и учёт IT-проектов, управление задачами, отчётность, разграничение доступа по ролям.

**Внешние системы (System_Ext):**

- **Корпоративная система аутентификации** — LDAP/AD: проверка логина/пароля и получение ролей пользователей.
- **Офисные приложения** — MS Word, MS Excel, просмотр PDF: открытие сформированных системой отчётных файлов (DOCX/XLSX/PDF).

Связи: все четыре актора взаимодействуют с ИС УПЗ IT по веб-интерфейсу (HTTPS); система обращается к системе аутентификации (LDAP/LDAPS) и отдаёт файлы отчётов для открытия в офисных приложениях.

## 4. Диаграмма контейнеров

**[Диаграмма контейнеров](containers.png)**

**Выбранный технологический стек:** backend — **C# / ASP.NET Core** (REST API, Entity Framework Core, формирование отчётов); фронтенд — **SPA на React** (JavaScript/TypeScript, HTML/CSS); база данных — **PostgreSQL**. Развёртывание — Docker-контейнеры.

### 4.1. Краткое пояснение по элементам

- **Веб-приложение (контейнер)** — одностраничное приложение (SPA) на React: интерфейс пользователя (формы, списки, отчёты, авторизация). Обращается к API Backend по HTTPS/JSON.
- **API Backend (контейнер)** — серверное приложение на **ASP.NET Core**: REST API, бизнес-логика, авторизация (в т.ч. ASP.NET Core Identity), CRUD, генерация отчётов (Word/Excel/PDF). Отдельный модуль развёртывания; общается с SPA по HTTPS/JSON и с БД через Entity Framework Core (протокол СУБД/TCP).
- **База данных (контейнер)** — **PostgreSQL**: хранение проектов, задач, пользователей, спринтов, справочников. Отдельный контейнер/сервис.

### 4.2. Выбор архитектурного стиля и топологии

**Стиль уровня приложений:** многоуровневая архитектура (N-tier / Layered) с раздельными модулями развёртывания:

- **Уровень представления:** SPA на React (клиентское приложение в браузере).
- **Уровень бизнес-логики и API:** API Backend на C# / ASP.NET Core (контроллеры, сервисы, EF Core).
- **Уровень данных:** PostgreSQL.

**Топология:** несколько модулей развёртывания и сетевое взаимодействие:

- Раздельные контейнеры: веб-приложение (React SPA), API Backend (ASP.NET Core), БД (PostgreSQL).
- Связи по сети: пользователь ↔ SPA (HTTPS), SPA ↔ API (HTTPS/JSON), API ↔ БД (EF Core / драйвер PostgreSQL).

## 5. Диаграмма компонентов

**[Диаграмма компонентов](components.png)**

Детализирован контейнер **API Backend** на стеке **C# / ASP.NET Core**.

### 5.1. Компоненты контейнера API Backend

- **AuthController, ProjectController, TaskController, UserController, SprintController, ReportController** — компоненты на **ASP.NET Core Web API**: приём HTTP-запросов, маршрутизация, вызов сервисов; вход/выход, CRUD проектов, задач, пользователей, спринтов, запрос отчётов с выбором формата (DOCX/XLSX/PDF).
- **ReportService** (C#) — формирование пяти отчётов по ТЗ, агрегация данных; экспорт в Word/Excel/PDF с использованием **Open XML SDK**, **EPPlus**, **QuestPDF** (или аналогов).
- **AuthService** (C#) — проверка логина/пароля, роли, разграничение доступа; опора на **ASP.NET Core Identity** при необходимости.
- **ProjectService**, **TaskService** (C#) — бизнес-логика проектов и задач, валидация, назначения и привязки.
- **DbContext / репозитории** (**Entity Framework Core**) — доступ к БД PostgreSQL: сущности проектов, задач, пользователей, спринтов, справочников.

### 5.2. Пояснение по новым элементам

Контроллеры ASP.NET Core принимают HTTP-запросы от клиента (React SPA) и делегируют выполнение сервисам. Сервисы (C#) реализуют бизнес-правила и расчёты; ReportService формирует файлы отчётов в форматах DOCX, XLSX, PDF. Слой доступа к данным (EF Core, репозитории) инкапсулирует работу с PostgreSQL. Внешний контейнер «База данных» — экземпляр PostgreSQL, с которым взаимодействует только этот слой.

## 6. Диаграмма последовательности (sequence)

[Диаграмма последовательности](sequence.png)

### 6.1. Соответствие элементам C4 и охват одного актора

- **Объекты**: актор (Руководитель проекта), контейнер «Веб-приложение», контейнер «База данных»; компоненты внутри API Backend — ReportController, ReportService, Repository.
- Рассмотрен **один актор** — Руководитель проекта.
- **Сценарий:** формирование отчёта «Статусы IT-проектов» в формате PDF.

Последовательность: пользователь выбирает отчёт и формат в веб-приложении → запрос к ReportController → вызов ReportService → ReportService запрашивает данные через Repository у БД → расчёт метрик и формирование PDF → возврат файла через API в веб-приложение → скачивание отчёта пользователем.

## 7. Диаграмма классов (class)

[Диаграмма классов](class.png)

### 7.1. Структура иерархии и покрытие функционала

- **Project** — IT-проект: код, наименование, ответственный (User), даты релиза. Связь с задачами и спринтами. Методы доступа и изменения полей.
- **Task** — задача: название, тип (фича/баг/техническая), приоритет, статус, срок, проект, спринт, исполнитель. Методы `isOverdue()` и `getDaysOverdue()` для отчёта по просроченным задачам.
- **User** — пользователь: логин, хеш пароля, ФИО, роль. Метод проверки пароля; связь с назначенными задачами и проектами (ответственный).
- **Sprint** — спринт/итерация: название, даты, проект; связь с задачами.
- **Status**, **Priority** — справочники статусов и приоритетов; связь с задачами.
- **TaskType**, **Role** — перечисления: тип задачи и роль пользователя.

Связи (агрегация/ассоциация) отражают: проект содержит задачи и спринты; пользователь назначен на задачи и является ответственным по проектам; задача связана с проектом, спринтом, статусом, приоритетом, исполнителем.

### 7.2. Краткое пояснение по классам

| Класс                    | Назначение                                                                                             |
| ------------------------ | ------------------------------------------------------------------------------------------------------ |
| **Project**              | Сущность IT-проекта для справочника проектов и отчётов (сводный по проектам, статусы проектов).        |
| **Task**                 | Задача разработки; поддержка типов, статусов, приоритетов, назначения и вычисляемых полей для отчётов. |
| **User**                 | Учётная запись и роль для авторизации и отчётов (исполнитель, ответственный).                          |
| **Sprint**               | Итерация/спринт для привязки задач и отчёта по загрузке команды с фильтром по спринту.                 |
| **Status**, **Priority** | Справочники для задач и вывода в отчётах «название статуса/приоритета».                                |
| **TaskType**, **Role**   | Перечисления для типа задачи и роли доступа.                                                           |

- В отчёте зафиксирован выбранный технологический стек: backend — C# / ASP.NET Core, фронтенд — SPA на React, БД — PostgreSQL.

**Плюсы:**

- Единый набор диаграмм (UML + C4) даёт согласованное представление о системе от контекста до компонентов и сценариев.

- Диаграмма классов описывает доменную модель; слой контроллеров/сервисов и DTO на диаграмме классов не раскрыт.

**Реализовано по заданию:** постановка задачи, функциональные требования, use case, C4 контекст и контейнеры с пояснением стиля и топологии, диаграмма компонентов для одного контейнера, диаграмма последовательности для одного актора, диаграмма классов с пояснениями, вывод.

**Дальнейшие действия:** использование результатов проектирования на этапах проектирования БД (нормализация таблиц по сущностям из диаграммы классов), реализации на выбранном стеке (ASP.NET Core Web API, React SPA, Entity Framework Core, генерация отчётов Open XML/EPPlus/QuestPDF), тестирования и развёртывания в Docker.
